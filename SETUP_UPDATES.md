# Auto-Updates Setup Guide

This guide explains how to configure the auto-update system for the Video Converter application using GitHub Releases.

## Overview

The application uses `electron-updater` to automatically check for and download updates from GitHub Releases. The auto-update system:

- ✅ Checks for updates automatically every 4 hours (production only)
- ✅ Allows manual update checks via "Check for Updates" button in Settings
- ✅ Downloads updates in the background with progress tracking
- ✅ Notifies users when updates are available and ready to install
- ✅ Validates configuration before making API calls to GitHub

## ⚠️ Configuration Required

Before auto-updates will work, you need to update the GitHub repository information in **two places**:

### 1. Update `src/main/autoUpdater.ts` (Lines 9-10)

Replace the placeholder values with your actual GitHub repository:

```typescript
// Configuration constants
const GITHUB_OWNER = 'your-github-username'; // ← Change this to your GitHub username
const GITHUB_REPO = 'video-converter';       // ← Change this to your repository name
```

**Example:**
```typescript
const GITHUB_OWNER = 'johndoe';
const GITHUB_REPO = 'my-video-converter';
```

### 2. Update `forge.config.js` (Lines 80-86)

Update the publishers section:

```javascript
publishers: [
  {
    name: '@electron-forge/publisher-github',
    config: {
      repository: {
        owner: 'your-github-username', // ← Change this
        name: 'video-converter'        // ← Change this if needed
      },
      prerelease: false,
      draft: true
    }
  }
]
```

### 3. Update `package.json` (repository field - optional)

```json
"repository": {
  "type": "git",
  "url": "https://github.com/your-github-username/video-converter.git"
}
```

### 4. Rebuild the Application

After updating the configuration:

```bash
npm run build
```

## Testing Auto-Updates

### Local Testing (Development Mode)

When running in development mode with `npx electron .`:

1. Auto-updater is **initialized** but periodic checks are **disabled**
2. You can manually click "Check for Updates" in Settings
3. If configuration not updated, you'll see the friendly validation message
4. Once configured, it will check GitHub but show "No updates available" in dev mode

### Production Testing

1. **Package the application:**
   ```bash
   npm run make
   ```

2. **Install the packaged app:**
   - **macOS:** `out/make/zip/darwin/x64/` - Extract and move to Applications
   - **Windows:** `out/make/squirrel.windows/x64/` - Run Setup.exe
   - **Linux:** `out/make/deb/x64/` - Install .deb package

3. **Run the installed app** (not from terminal)

4. **Test update flow:**
   - Open Settings tab
   - Click "Check for Updates"
   - Should see either validation message, "No updates available", or update dialog

## Publishing Releases

### Prerequisites

1. **GitHub Repository:** Create a public or private repository

2. **GitHub Token:** For publishing (optional - can upload manually)
   ```bash
   # Create token at: https://github.com/settings/tokens
   # Scope: repo (full control)
   export GITHUB_TOKEN=your_token_here
   ```

3. **Version Updated:** Edit `package.json`:
   ```json
   {
     "version": "1.1.0"
   }
   ```

### Publishing Workflow

**Option 1: Automatic Publishing (with GITHUB_TOKEN)**

```bash
# Build and package
npm run build && npm run make

# Publish to GitHub (creates draft release)
npm run publish

# Go to GitHub → Releases → Edit draft → Publish
```

**Option 2: Manual Upload**

```bash
# Build and package
npm run build && npm run make

# Create release on GitHub manually:
# 1. Go to your repository → Releases → Create new release
# 2. Tag: v1.1.0 (must match package.json version)
# 3. Title: v1.1.0 or "Version 1.1.0"
# 4. Upload files from out/make/:
#    - macOS: video-converter-darwin-x64-1.1.0.zip
#    - Windows: All files from squirrel.windows folder
#    - Linux: video-converter_1.1.0_amd64.deb
# 5. Add release notes
# 6. Publish release
```

### Required Release Files

electron-updater looks for specific files:

- **macOS:** `.zip` file + `latest-mac.yml` (auto-generated by electron-builder)
- **Windows:** `.exe` installer + `RELEASES` file + `.nupkg` files  
- **Linux:** `.deb` or `.AppImage` + `latest-linux.yml`

## Update Flow (User Experience)

### Automatic Check (Every 4 Hours - Production Only)

1. App runs in background timer
2. Checks GitHub for new releases
3. If new version found → Shows "Update Available" dialog
4. User decides: Download now or Later

### Manual Check (Settings Button)

1. User clicks "Check for Updates" in Settings
2. Shows spinner "Checking..."
3. Results:
   - **Not configured:** Friendly message with setup instructions
   - **No update:** Snackbar: "No updates available"
   - **Update found:** "Update Available" dialog appears

### Download Phase

1. User clicks "Download Update"
2. Progress bar shows:
   - Download percentage (0-100%)
   - Transfer speed (MB/s)
   - Downloaded / Total size
3. Download happens in background (app stays usable)

### Install Phase

1. Download completes → "Update Ready" dialog  
2. User clicks "Restart & Install"
3. App quits automatically
4. Update installs
5. App relaunches with new version

## Error Handling

The app provides user-friendly error messages:

### Configuration Errors

**Placeholders not replaced:**
```
Auto-updates not configured. Please update GITHUB_OWNER and 
GITHUB_REPO in autoUpdater.ts with your GitHub repository details.
```

**Repository not found (404):**
```
GitHub repository not found or has no releases. Please check 
your repository configuration in autoUpdater.ts
```

### Network/Release Errors

**No internet connection:**
```
Failed to check for updates: Network error
```

**No releases published:**
```
GitHub repository not found or has no releases.
```

All errors appear as dismissible notifications in bottom-left corner for 6 seconds.

## Validation Logic

The app validates configuration before contacting GitHub:

```typescript
// src/main/autoUpdater.ts (lines 12-14)
function isUpdateConfigured(): boolean {
  return GITHUB_OWNER !== 'your-github-username' && 
         GITHUB_REPO !== 'video-converter';
}
```

This prevents:
- 404 errors from hitting GitHub with placeholder values
- Unnecessary API requests during development
- Confusing error messages for developers

## Troubleshooting

### "Update error: [object Object]" in Console

This console output is normal - the actual user-facing error message appears as a Snackbar notification in the app UI (bottom-left corner). The Snackbar shows the friendly error message.

### No Updates Found (Despite New Release)

**Check these:**
1. ✅ Release is **published** (not draft) on GitHub
2. ✅ Release tag format: `v1.1.0` (with 'v' prefix)
3. ✅ Release version is **higher** than current app version
4. ✅ Platform-specific files are attached to release
5. ✅ Repository is public OR GitHub token configured for private repo

### 404 Errors from GitHub API  

**Before fix:** App would hit GitHub API with placeholder values causing 404s

**After fix (current):** Validation prevents API calls, shows friendly message instead

If you still see 404s after configuring:
- Double-check GITHUB_OWNER and GITHUB_REPO values
- Verify repository exists and is accessible
- Check repository name spelling (case-sensitive)

### Download Not Starting

**Possible causes:**
- No internet connection
- GitHub is down
- Release files not properly uploaded
- Insufficient disk space

**Check logs:**
- Open DevTools (Cmd+Option+I / Ctrl+Shift+I)
- Check Console tab for electron-updater logs

## Reverting Test Configuration

Currently, auto-updater is enabled in development for testing. To revert to production-only:

Edit [`src/main/main.ts`](src/main/main.ts) lines 80-87:

```typescript
// Initialize auto-updater (only in production)
if (mainWindow && process.env.NODE_ENV !== 'development') {
  updater = new AutoUpdater(mainWindow);
  updater.startPeriodicChecks(4); // Check every 4 hours
}
```

Then rebuild:
```bash
npm run build
```

## Security & Best Practices

1. **Code Signing:** Sign your builds for production (prevents security warnings)
2. **HTTPS Only:** electron-updater enforces HTTPS (built-in)
3. **Private Repos:** Set `GH_TOKEN` env variable for private repository access
4. **Test Thoroughly:** Always test updates on clean install before publishing
5. **Backup:** Keep backups of working builds before releasing updates
